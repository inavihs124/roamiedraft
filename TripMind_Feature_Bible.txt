#01
Module: Itinerary
Live Itinerary Engine
Your trip plan is never frozen — it reacts to the real world in real time.
W H A T  I T  D O E S
Subscribes to live data feeds — weather 
forecasts, venue opening hours, flight 
status updates. When any real-world 
condition changes (rain forecast, venue 
closes early, flight delayed), the itinerary 
automatically patches the affected 
events without the user doing anything.
U S E R  I M P A C T
Eliminates the moment every traveller 
knows: arriving at a place to find it's 
closed, or stepping outside into rain you 
didn't plan for. The plan is always 
current. Zero manual update required.
H O W  T O  B U I L D
1. On itinerary load, register background 
polling for: Open-Meteo (weather, every 
3h), Overpass API (venue hours, 
once/day), flight status (every 30min).
2. On any change, compare with stored 
ItineraryDay objects.
3. If mismatch → call Ollama LLM with 
updated context → regenerate affected 
events.
4. Push updated itinerary to UI via 
WebSocket or polling.
S T A C K   /   A P I s   
→  
Open-Meteo (weather, free, no key)  ·  Overpass API (venue data, free)  ·  OpenSky Network (flight tracking, free)  ·  
Ollama LLM (local, offline)

#02
Module: Itinerary
Itinerary Auto-Healer
One disruption. Full cascade fix. Flight → hotel → cab → schedule. All automatic.
W H A T  I T  D O E S
When a disruption is detected (flight 
cancelled, delayed, missed), the system 
does not just flag the broken event. It 
identifies every downstream event that 
is now invalid — hotel check-in time, 
pre-booked cab, restaurant booking, day 
activities — and replans the entire 
affected timeline from scratch.
U S E R  I M P A C T
Before TripMind: flight cancelled → 45 
minutes of panicking to manually fix 6 
things. After TripMind: one notification 
saying 'We've fixed everything. Here's 
your new plan.' The emotional cost of 
disruption drops to near zero.
H O W  T O  B U I L D
1. Disruption event triggers 
TriggerDisruptionShield use-case.
2. Load all ItineraryDay records for 
affected dates.
3. Identify all downstream bookings 
(hotel, cab, activities) with timestamps 
after the disruption.
4. For each affected booking: calculate 
new time = disruption_resolution_time + 
buffer.
5. Call Ollama with new constraints to 
rebuild ItineraryDay JSON.
6. Save updated records to DB + push 
to UI.
S T A C K   /   A P I s   
→  
Internal cascading use-case logic  ·  Prisma DB (SQLite/Postgres)  ·  Ollama LLM (local)  ·  Mock flight data JSON (30 
flights, 6 routes)

#03
Module: Itinerary
Calendar Gap Detector
Finds your free time. Fills it with exactly what fits — automatically.
W H A T  I T  D O E S
Reads the user's calendar events 
(pasted or synced). Detects every gap 
between 30 and 90 minutes that falls 
between scheduled events. For each 
gap, queries nearby places (food, 
activity, rest) filtered by current GPS 
location, time of day, and trip purpose 
(business vs leisure). Pushes 3 ranked 
suggestions directly into the itinerary 
timeline.
U S E R  I M P A C T
Business travellers stop wasting lunch 
hours sitting in hotel rooms. Leisure 
travellers discover local spots they 
would never have found. The app 
becomes a proactive assistant, not a 
passive calendar.
H O W  T O  B U I L D
1. Parse calendarEvents[] array, sort by 
start time.
2. Iterate pairs: if gap between 
event[n].end and event[n+1].start is 30–
90min → store as FreeGap.
3. For each FreeGap: call Foursquare 
Places API (or Overpass) with lat/lng + 
radius 1km + category filter.
4. Pass gap duration + trip purpose to 
Ollama → returns 3 ranked suggestions.
5. Inject into ItineraryDay.freeGaps[] and 
render with amber left-border in UI.
S T A C K   /   A P I s   
→  
Calendar API (Google Calendar free tier) or manual paste  ·  Foursquare Places API (1000 calls/day free)  ·  Overpass 
API (fallback, free)  ·  Ollama LLM

Module: Disruption
Auto Flight Alternative Holder
Disruption detected → 3 alternatives held → best one pre-selected → you just confirm.
W H A T  I T  D O E S
The moment a disruption is confirmed, 
the system queries the flight dataset for 
alternatives on the same route within ±6 
hours of the original departure. Returns 
exactly 3 options, each scored by a 
weighted formula: 40% weight on 
earliest arrival time (most important —
time is precious during disruption), 30% 
on price difference from original, 30% on 
whether the seat type matches user 
preference (aisle/window/business). The 
best-scored option is pre-selected. User 
confirms or chooses another.
U S E R  I M P A C T
Compresses the most stressful 45 
minutes of any trip into a 10-second 
decision. You don't research, you don't 
search, you don't compare. Three pre-
evaluated options appear. Tap the one 
you want. Done.
H O W  T O  B U I L D
1. On disruption trigger: call 
MockFlightService.findAlternatives(origi
n, dest, date, maxResults: 3).
2. MockFlightService queries static 
JSON of 30 flights filtered by route + 
time window.
3. For each alternative: compute score = 
(0.4 × arrivalEarliness) + (0.3 × (1 -
priceDelta/originalPrice)) + (0.3 ×
seatMatch).
4. Sort by score desc → return top 3.
5. Pre-select index[0] as recommended.
6. Render 3 flight cards in UI — selected 
card gets amber border + 'Best match' 
S T A C K   /   A P I s   
→  
MockFlightService (static JSON, 30 flights, 6 routes — no API needed for demo)  ·  AviationStack API (real data, 100 
free calls/month — use for live demo if available)
